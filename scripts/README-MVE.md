# MVE Quick Start

7-day experiment to test the phone-first capture premise before committing to the alphaclaw migration. See PRD `state/prd-alphaclaw-migration.md` §9.-1.

## What you need to do manually (one-time, ~15 min)

### 1. Create Telegram bot
- On phone, message **@BotFather** in Telegram.
- Send `/newbot`. Pick a name (e.g., "Aaron COO Twin MVE") and a username ending in `bot` (e.g., `aaron_coo_mve_bot`).
- BotFather returns a token like `123456:ABC-DEF...`. **Save it** — this is `TELEGRAM_BOT_TOKEN`.
- Send `/setprivacy` to BotFather, choose your bot, select **Disable** (so it can read all messages in chats it's added to, not just commands).

### 2. Get your chat ID
- Message **@userinfobot** in Telegram. It replies with your user ID (numeric, ~9 digits). Save it — this is `TELEGRAM_CHAT_ID`.
- Start a DM with your new bot (search its username, hit Start).

### 3. Save secrets
Create `C:\Users\aaron\daily-automation\.secrets\telegram.env`:

```
$env:TELEGRAM_BOT_TOKEN = "123456:ABC-DEF..."
$env:TELEGRAM_CHAT_ID = "987654321"
```

(File is already gitignored via `.secrets/`.)

### 4. Test send
Open PowerShell in the repo root:

```powershell
. .\.secrets\telegram.env
"hello from MVE" | .\scripts\mve-telegram-send.ps1
```

You should get a Telegram DM from your bot within ~2 seconds.

### 5. Test receive
With the bot DM open on phone, send any text message. Then run:

```powershell
. .\.secrets\telegram.env
.\scripts\mve-telegram-receive.ps1
```

You should see `[q-001] type=text len=N` and a new file at `inbox/q-001.json`. Phone should get a reply "Queued #q-001..."

### 6. Schedule daily briefing (laptop-side Telegram delivery)

The morning briefing is generated by either:
- Cowork's `morning-briefing-cowork` scheduled task (cloud, runs even when laptop is off) — writes to `vault/daily/YYYY-MM-DD.md` and mirrors to the "📱 Daily Briefing — Mobile" Notion page (`mobile_briefing_page_id` in `config/sources.yaml`). Phone-read via Notion mobile app.
- Local Claude Code `/start-day` run on the laptop — same file output + same Notion mirror.

Telegram delivery is a **second** phone-read surface. Cowork can't reach your Telegram secrets, so the Telegram send fires from your laptop only.

**Task Scheduler entry — Create Basic Task:**
- Name: `MVE - start-day Telegram send`
- Trigger: Daily at 7:00 AM
- Action: Start a program
- Program: `pwsh.exe`
- Arguments (verbatim — single line):
  ```
  -NoProfile -ExecutionPolicy Bypass -Command ". .\.secrets\telegram.env; Get-Content -Raw \"vault\daily\$(Get-Date -Format yyyy-MM-dd).md\" | .\scripts\mve-telegram-send.ps1"
  ```
- Start in: `C:\Users\aaron\daily-automation`
- Conditions tab: uncheck "Start the task only if the computer is on AC power" (so it fires on battery too).
- Settings tab: check "Run task as soon as possible after a scheduled start is missed."

The Get-Content reads whatever `/start-day` (Cowork or local) wrote to the daily-note file. If neither ran yet by 7am the read fails — Task Scheduler logs the error, no message sent. Day-1 fix: just run `/start-day` manually in Claude Code before 7am, or trust the 6am Cowork morning run to write the file first.

**Smoke test after BotFather + step 3 (no Task Scheduler needed):**
```powershell
. .\.secrets\telegram.env
"MVE smoke test $(Get-Date -Format 'HH:mm')" | .\scripts\mve-telegram-send.ps1
```
You should see a Telegram DM within ~2 seconds. If `Missing TELEGRAM_BOT_TOKEN` errors: the env file didn't source. If `Telegram returned ok=false`: token or chat ID wrong.

**Full daily-note smoke test (validates Task Scheduler arguments without scheduling):**
```powershell
. .\.secrets\telegram.env
Get-Content -Raw "vault\daily\$(Get-Date -Format yyyy-MM-dd).md" | .\scripts\mve-telegram-send.ps1
```
Should deliver the entire daily note in chunks of 3800 chars.

### 7. Optional: continuous inbox polling
Run in a separate terminal during the week:

```powershell
. .\.secrets\telegram.env
.\scripts\mve-telegram-receive.ps1 -Loop -PollSeconds 30
```

Or schedule another Task Scheduler entry to run `mve-telegram-receive.ps1` (one-shot) every 5 min.

## What you do each day (~30 sec)

- Phone gets briefing at 7am. Read it (or don't — both are signal).
- When something capture-worthy happens during the day: DM the bot. Type a sentence, paste a Granola URL, whatever.
- At day's end: open `logs/MVE-bypass.md`, fill in the day's row. **Be honest about what you didn't capture.**

## Drain the inbox

When you're at the laptop and ready to process queued captures:

```powershell
# In Claude Code
/capture-meeting
```

Have /capture-meeting check `inbox/*.json` first. Process each, then move to `inbox/processed/`.

(For MVE we won't build the inbox-aware capture-meeting variant yet — just manually point Claude at the inbox JSON files. If MVE passes, that integration is part of full-migration Phase 4.)

## After 7 days

Fill the Week Summary at the bottom of `logs/MVE-bypass.md`. Decision per §9.-1:
- **≥30% bypass drop + briefings read** → proceed to full migration (G1 gate, then Phase 0 Render setup)
- **Briefings read but bypass flat** → ship MVE as permanent, skip alphaclaw entirely (massive scope reduction)
- **Briefings ignored AND bypass flat** → kill the migration thesis; rethink

The whole point is to have data before committing 8–11 weeks of work.
